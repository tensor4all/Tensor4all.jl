**Verdict:** fail

## Per-Feature Details

### CONTRIBUTING.md flow
- **Role:** Physics PhD student, intermediate Julia, new to project
- **What they tried:** Read README → CONTRIBUTING.md → understand the flow
- **Discoverability:** Good. README links to CONTRIBUTING.md. Flow diagram is clear.
- **Functionality:** Works as documentation. Label system is well-explained.
- **Friction points:** No explanation of how to actually use AI skills.

### Issue templates
- **What they tried:** Read `.github/ISSUE_TEMPLATE/` files
- **Discoverability:** Good. CONTRIBUTING.md points to them.
- **Functionality:** Well-structured. Feature Request maps cleanly to spec-proposal output format.
- **Friction points:** Templates set `enhancement`/`bug` labels but not `proposed`, which CONTRIBUTING.md defines as the initial state.

### AI skills (spec-proposal, plan-proposal)
- **What they tried:** Look for skills in Claude Code, try to understand invocation
- **Discoverability:** CONTRIBUTING.md mentions skills exist in `.claude/skills/`
- **Setup:** **CRITICAL** — Skills are `.claude/skills/<name>/SKILL.md` files. This format is discoverable by OpenCode but NOT automatically registered as Claude Code slash commands. Claude Code uses `.claude/commands/<name>.md` for custom slash commands.
- **Functionality:** Skill content (output format, guidelines) is adequate.
- **Blocked steps:** Cannot invoke `/spec-proposal` in Claude Code.
- **Friction points:** No usage instructions. Contributor doesn't know if these are slash commands, prompts to paste, or templates to read.

### Proactive guidance (AGENTS.md)
- **What they tried:** Evaluate whether the "proactive guidance" section works
- **Discoverability:** Good. AGENTS.md clearly lists trigger patterns and response templates.
- **Functionality:** The chain breaks: AI suggests "use spec-proposal skill" → but cannot invoke it as a registered skill → AI would need to manually read SKILL.md and follow it.

## Expected vs Actual Outcome

**Expected:** Contributor says "TensorTrain doesn't support addition" → AI proactively suggests spec-proposal → AI invokes skill → spec is drafted and posted as issue.

**Actual:** The guidance text in AGENTS.md works, but the mechanical invocation fails because `.claude/skills/` is OpenCode's format, not Claude Code's custom command format. The AI would need to manually read the SKILL.md file as a prompt, which works but is undocumented and fragile.

## Issues Found

### Critical

1. **Skills not registered for Claude Code.** Claude Code custom commands use `.claude/commands/<name>.md` (flat files, no frontmatter). The current `.claude/skills/<name>/SKILL.md` format is OpenCode-compatible but not Claude Code-invocable. Need to either:
   - Add `.claude/commands/` aliases that invoke the skills, or
   - Document that skills are prompt templates to be read manually

2. **Issue templates missing `proposed` label.** CONTRIBUTING.md defines `proposed` as the initial state, but `feature_request.yml` sets `enhancement` and `bug_report.yml` sets `bug`. The `proposed` label is never applied automatically.

### Non-critical

3. **No "how to use skills" section in CONTRIBUTING.md.** Contributors don't know the invocation mechanism.
4. **README "Still Missing" is too vague.** Does not list specific missing operations (TensorTrain arithmetic, norm, etc.), making it hard for contributors to identify what to work on.
5. **Documenter.jl site has no mention of contribution flow.** A user starting from the docs site misses the entire workflow.

## Suggestions

1. **Add `.claude/commands/` entries** for Claude Code compatibility — either as full copies or as thin wrappers that read the SKILL.md.
2. **Add `proposed` to issue template labels** alongside `enhancement`/`bug`.
3. **Add "How to use skills" section** to CONTRIBUTING.md with concrete invocation examples for Claude Code and OpenCode.
4. **Enumerate missing operations** in README.md "Still Missing" section.
5. **Add contribution flow link** to `docs/src/index.md`.
